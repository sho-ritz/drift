## 私はドキュメントを腐らせた側の人間です

最初に白状します。この記事は、ドキュメントを腐らせた被害者ではなく、**加害者**が書いています。

仕事でチーム開発を始めたとき、私たちは実装を急ぎました。機能を出すことが最優先で、ドキュメントの更新は「あとで」に積まれ続けた。数ヶ月後に気づいたときには、仕様書は現実のコードと乖離しすぎていて、**もう直す気すら起きない、読む価値のない何か**になっていました。誰かが悪意で放置したわけではありません。全員が真面目に開発していて、それでもこうなった。

次のプロジェクトでは反省してNotionでドキュメントを整備しました。今度はAIエージェントの時代です。Notion MCPをつないで、エージェントに「この仕様に対応するコードはどこ?」と調べさせる運用を試しました。そして二つの現実に直面します。

一つ、**Notion MCPは、この用途には機能が足りない**。ページの取得や検索はできても、「この見出しのセクションとこの関数が対応している」という関係を扱う語彙がそもそも存在しない。対応関係の調査は毎回、検索とページ読み込みの繰り返しになる。

二つ、**トークン消費が激しすぎる**。外部SaaSをMCP越しに読むオーバーヘッドは想像以上で、ページ構造のメタデータごと大量のコンテキストを食い、それを何往復も繰り返す。「仕様を1セクション確認する」だけのために、リポジトリ内のファイルを読む何倍ものトークンが溶けていきました。

この二つの実体験 — **リポジトリ内でも規律だけでは腐る。リポジトリ外に出すとエージェントから遠すぎる** — が、この記事で語るツールの出発点です。

## ドキュメントが腐る本当の理由は、書き手の怠慢ではない

コードは秒で変わる時代になりました。AIエージェントが実装し、リファクタリングし、テストを書く。その加速の中で、腐ったドキュメントは単なる不便では済みません。**エージェントは古い仕様書を「正」として読み、すでに存在しない挙動に合わせたコードを再生産します。** 人間なら「これ古くない?」と感じる違和感を、エージェントは持ちません。ドキュメントの腐敗は、開発パイプラインに毒を流すインシデントです。

この問題は昔から「書き手の規律」の問題として語られてきました。「PRを出すときはドキュメントも更新しましょう」。私はもうこの説明を信じていません。かつての私のチームも、ルールがなかったから腐らせたのではない。ルールがあっても、締切の前で必ず崩壊した。

何度も失敗して辿り着いた私の結論はこうです。

**ドキュメントが腐る根本原因は、ドキュメントがコードと別の場所に住んでいるからです。**

仕様書はNotionに、設計メモはConfluenceに、コードはGitHubに。この分断こそが諸悪の根源です。

- コードの変更はPRでレビューされるのに、**仕様の変更は誰のレビューも通らずに上書きされる**(あるいは誰にも編集されず化石になる)
- コードには歴史(git log)があるのに、**仕様には「いつ、誰が、なぜ」の記録がない**
- コードの世界とドキュメントの世界に**リンクを張る足場が存在しない**。NotionのURLはただの文字列で、リネーム一つで死ぬ
- そして決定的なのは、**AIエージェントはリポジトリの中で働く**ということ。リポジトリの外の情報は、エージェントにとって「遠くて高くて信頼できない」— 私がNotion MCPで身をもって学んだ通りです

距離が腐敗を生む。だから私の答えはシンプルです。

## 思想: ドキュメントとコードは、同じリポジトリに住むべきだ

私が作った [Drift](https://github.com/sho-ritz/drift) の根底にあるのは、この一つの思想です。**仕様書をコードと同じリポジトリの `/docs` に置き、同じgitの規律で管理する。** ドキュメント基盤をSaaSに外注するのをやめて、リポジトリに帰す。

先に認めておくと、「ドキュメントをリポジトリに置く」という考え自体は新しくありません。Docs as Code は10年以上前からある実践で、ADR(Architecture Decision Records)もその系譜です。**しかし、置くだけでは腐ります。** 私の最初のチームがまさにそうでした — Markdownはリポジトリの中にあったのに、コードとの対応は誰も追跡しておらず、リポジトリの中で静かに化石になった。あなたも、git管理されているのに古いままのREADMEをいくつも知っているはずです。

つまり、この10年のDocs as Codeに欠けていたのは置き場所ではなく、**対応関係の強制装置**です。統合には明確な果実がある —

1. **一つの歴史。** 仕様の変更もコードの変更も同じコミットグラフに乗る。仕様にblameが効く。
2. **一つのレビュー。** 仕様の書き換えがPRとして流れてくる。コードと同じ場所で、同じ緊張感でレビューされる。
3. **一つのコンテキスト。** エージェントに「リポジトリを読め」と言えば、コードも仕様も同じ視界に入る。MCP越しの外部SaaS往復も、その莫大なトークンも消える。
4. **リンクを張る足場。** 同じファイルシステム、同じバージョン管理の上にあるからこそ、「この関数はこのセクションに対応する」という**機械的に検証可能なエッジ**を張れる。NotionとGitHubの間には原理的に張れません。

— が、その果実は4番の「エッジ」を実際に張り、**張ったエッジの鮮度を機械が守り続けて**初めて収穫できます。統合が土台、証明がエンジン。Driftは、このエンジンを作るプロジェクトです。

> **[画像2: 思想の図解]**
> 対比の図。左半分「分断された世界」: Notion・Confluence・GitHubが別々の島として描かれ、島の間に切れたリンク(破線と?マーク)、Notionの島とエージェントの間に「MCP経由・高コスト」のラベル。右半分「統合された世界」: 一つのリポジトリの箱の中に `src/` と `docs/` が並び、その間に実線のエッジ(Link)が張られ、エッジに「hash ✓」のラベル。左→右への大きな矢印。この記事の主張を一枚で表す最重要の図。

## Drift — 「たぶん合ってる」を「変わっていないと証明できる」に変える

統合されたリポジトリの上で、Driftはコードのシンボル(関数・クラス)と仕様書のセクションを**リンク**として結び、リンクを承認した瞬間の**両側のハッシュをピン留め**します。以後の状態は3値です。

- ✅ **fresh** — 承認時からコードも仕様も変わっていない
- ⚠️ **stale** — どちらかが編集された。乖離したかもしれない
- ❌ **broken** — シンボルかセクションが消えた

ここで、言葉の範囲を正確にしておきます。**ハッシュが証明するのは「変わっていないこと」であって、「正しいこと」ではありません。** freshは「最後に人間(またはエージェント)が正しいと承認した状態から、両側とも無変更である」ことの証明です。変わった後にまだ正しいかどうかは、機械には決められない — それは判断の仕事です。

ではDriftの価値はどこにあるのか。**判断が必要な箇所を、判断が必要になった瞬間にだけ、確実に浮かび上がらせる**ことです。LLMに「このドキュメント全体、合ってる?」と聞けば、それらしい答えは返ってきます。しかしそれは全範囲に対する「たぶん」でしかない。Driftは範囲を絞ります: freshな部分は証明済みだから読み飛ばしてよい、staleなこの3箇所だけが判断を要する。**判定(変わったか)は機械に、判断(まだ正しいか)は知性に。** この分業がDriftの設計原理で、だから平常時のLLM呼び出しはゼロ、コストもゼロです。

> **[画像3: drift status のスクリーンショット]**
> fresh(緑✓)・stale(黄△)・broken(赤✗)が混ざった `drift status` の出力。drift リポジトリ自身で docs/design.md を1行編集してから `drift status` を実行すると stale が撮れる(撮影後 `git checkout docs/` で戻す)。カラー出力のまま撮ること。

### 仕組みは3つの部品だけ

```
SpecAnchor(仕様書のセクション) ←— Link(承認時ハッシュ入り) —→ Symbol(コードの関数・クラス)
```

1. **SpecAnchor** — Markdownの見出しから自動で切り出されるセクション(`docs/auth.md#login-flow`)。見出しを書くだけでアンカーになる。仕様の書き手に追加の作法を要求しないのは意図的です。**ドキュメントを書くこと自体のハードルを1ミリも上げてはいけない** — ハードルを上げた瞬間、書かれなくなることを私は知っています。
2. **Symbol** — コード側の関数・クラス。tree-sitter内蔵インデクサで解析するので外部ツール依存はゼロです。
3. **Link** — 両者を結ぶ多対多のエッジ。承認時点の正規化済みハッシュ(ASTベースでコメント・整形ノイズを除去)を持つので、インデント変更やコメント追加ではstaleになりません。

> **[画像4: アーキテクチャ図]**
> 左に Markdown ドキュメント(見出しごとに SpecAnchor として切り出される様子)、右にコードファイル(関数ごとに Symbol)、中央に Link のエッジが張られたグラフ図。各 Link に「approved hash: a3f2…」のようなラベルを付け、片側が変更されたエッジ1本だけ黄色く光らせると「検知」が伝わる。

リンクを張るコストにも執着しました。コメント1行(`// @spec: docs/auth.md#login-flow`)で張れる。コードに触りたくなければ外部の `links.json` でも張れる。そして既存コードベースには `drift suggest` — 仕様書内の `` `refreshToken` `` のようなコード言及や見出しとシンボル名の一致から、**LLMなしの決定的マッチング**で初期リンク候補を洗い出します。導入は「全手動アノテーション」ではなく「候補のレビュー」から始まる。**ビッグバン導入を要求するツールは必ず捨てられる**というのが、私がツール導入の失敗から学んだ教訓だからです。

## ある金曜日 — Driftのいる開発の一日

機能を列挙するより、一日の情景を見てもらう方が早いはずです。

金曜の夕方、エージェントに任せていた認証まわりのリファクタリングPRが上がってきます。差分は12ファイル。かつての私なら「このPR、どの仕様書に影響するんだっけ」を考えることすら放棄していたでしょう。今はPRに、botのコメントが1枚付いています。

> △ **Drift**: このPRが触ったリンクのうち2件が要確認です。
> ・stale: `AuthService::login` ↔ `docs/auth.md#login-flow`(code changed)
> ・broken: `refreshToken` ↔ `docs/auth.md#token-refresh`(symbol missing)

brokenの方は、ローカルで `drift status` を見ると続きがあります。

```
✗ [7] BROKEN (symbol missing)  refreshToken ↔ docs/auth.md#token-refresh
    ↳ moved? source now matches 'renewToken' (src/auth.ts) — drift relink 7
```

エージェントが関数名を変えていただけでした。`drift relink 7` — 3秒で修復。リネームのたびにリンクが死ぬツールなら、私はとっくに剥がしています。

staleの方は `drift verify` に判断させます。staleになったリンク**だけ**を対象に、Claudeが「現在のコードは現在の仕様文をまだ満たしているか」を判定する仕組みです。返ってきた判定は `diverged`。しかも、どの文と矛盾するかを逐語で引用してきます。

```
✗ [3] DIVERGED  AuthService::login ↔ docs/auth.md#login-flow
    contradicts: "ログイン成功時はセッショントークンを返す"
```

なるほど、トークン方式が変わったのに仕様が古い。仕様書を3行直して、`drift approve 3`。ここまで全部で2分です。CIの `drift check` が緑に戻り、**「このリポジトリの仕様とコードは、全リンクで一致が確認されている」という状態**でマージされます。

この一日の中で、LLMが呼ばれたのはverifyの1回だけ。ハッシュ判定は全部無料です。変化のない日、Driftは1トークンも消費しません。

> **[画像5: moved検知とrelinkのスクリーンショット]**
> 関数をリネーム→ `drift status`(brokenと↳moved?ヒント)→ `drift relink` → `drift status`(fresh復帰)の3連コマンドを1枚に。上の物語の実物として機能する。

## 非エンジニアを、リポジトリに招き入れる

「ドキュメントをリポジトリに統合する」と言うと、必ずこう返されます。**「PMやQAはgitを使えないから、結局Notionに戻るんだよ」**。

私はこれを、統合思想が乗り越えるべき最大の壁だと思っています。だからDriftは最初からObsidianプラグインを同梱しています。

仕様の書き手は、git管理された `docs/` をObsidianのvaultとして開くだけ。ノートを開けば、そのノートの各セクションを実装しているコードと鮮度バッジが並ぶ。コードが変わってstaleになったリンクは「Still correct」ボタンで再承認できる。書き終わったら**Publishボタン一つ**で、裏では `git add / commit / push` が走ります。

書き手はgitを知らないまま、gitの規律の中で書いている。**ツールの都合に人を合わせるのではなく、人の道具の中にリポジトリを持ち込む。** ドキュメントを外部SaaSに出してエンジニアの視界から消すのではなく、非エンジニアをリポジトリに招き入れる方向に倒す — これが統合を絵空事にしないための、私なりの答えです。

エンジニア側にも同じ思想で、VS Code拡張があります。リンクされた関数の真上にCodeLensで `spec: ✓ fresh docs/auth.md#login-flow`。**目に入る場所にないツールは習慣にならない**からです。

## 「本当に効くのか」— Drift自身がDriftで守られている

強い主張には証明が要ります。だから**DriftのリポジトリはDrift自身で管理されています**。設計判断はすべて `docs/design.md` に書かれ(もちろんコードと同じリポジトリです)、11のセクションがそれぞれの実装とリンクされ、CIは1本でもstale/brokenがあればビルドを落とします。

このドッグフーディングは、**開始初日に本物のバグを2つ捕まえました**。

1. **tree-sitterの字句解析トラップ** — `/<!--/` で始まる正規表現リテラルがレガシーHTMLコメントと誤解釈され、ファイルのパースが崩壊してシンボルが誤登録される。Drift自身の `anchors.ts` で起きていました。
2. **スラッグ乗っ取りバグ** — ドキュメント本文でアンカー上書き構文に*言及しただけ*で、次の見出しのIDが書き換わる。これもDrift自身のリンクが突然brokenになって発覚しました。

どちらも「自分のリンクが壊れた」ことが発見のトリガーです。ツールが自分の欠陥を自分の仕組みで検出する。「動くはず」ではなく「動いていることが検証され続けている」— この状態を自分のプロダクトに対して持てたことが、私がこの思想への確信を深めた瞬間でした。

もう一つ、これは手柄話ではなく**自分の足を撃った話**です。導入実験中、設定ミス(`docsDir: "."`)によってDriftは**node_modules配下の4,534個の無関係なMarkdownセクション**を仕様アンカーとして誤インデックスしました。완全に私のツールの設計不備です。だから同じ銃を封じました — 現在のDriftはこの設定を起動時に拒否します。

```
drift: error: docsDir: docsDir "." points at the repository root; indexing would
sweep up node_modules, READMEs, and vendored markdown.
```

失敗を仕組みで塞ぐ。ツール自身も、その思想で作られています。

> **[画像8: CIのスクリーンショット]**
> GitHub Actions の drift ワークフローが緑 + `drift check --check` の「11 fresh, 0 stale, 0 broken / 11 links」出力。「仕様とコードの一致がCIでゲートされている」ことが一目でわかる。

### エージェントにとっては、検索が参照に変わる

Notion MCP時代の私は、エージェントに仕様を確認させるたびに、検索→ページ取得→また検索の往復で大量のトークンを燃やしていました。統合リポジトリ+リンクグラフでは、これが一撃になります。

MCPツール `drift_context <symbol>` を呼ぶと、そのシンボルに対応する仕様セクションの**キャッシュ済み本文**が、鮮度フラグ付きで返ります。ドキュメント全体をgrepし直す必要も、外部SaaSと往復する必要もない。しかも返ってきた内容には「fresh = 承認時から無変更」という信頼度が付いている。

**探索がO(ドキュメント全文)からO(エッジ次数)になり、結果に信頼度が付く。** ドキュメントは「エージェントが読む散文」から「エージェントが参照するインフラ」に変わります。エージェント時代のドキュメントの最終形はこれだと、私は考えています。

## 5分で始める(そして、いつでもやめられる)

```sh
npm i -D driftdocs
npx drift init        # バックエンド・戦略を対話で選択
npx drift index       # docs/ から仕様アンカーを抽出
npx drift sync        # @spec: コメントとlinks.jsonを反映
npx drift status      # 全リンクの fresh/stale/broken
```

全部やる必要はありません。**まず1つのドキュメントと1つの関数をリンクするだけでいい。** Driftが触るのは `.drift/` ディレクトリとあなたが書いたコメント行だけなので、合わなければ消せば終わりです。退路のあるツールしか、私は人に勧めたくありません。

> **[画像9: drift init のスクリーンショット]**
> ASCIIアートバナー → 対話プロンプト → 「Drift is ready.」ボックスまでを1枚で。initの体験は「使ってみたい」を左右するので丁寧に撮る。

日常はこれだけです。

```sh
npx drift status --diff origin/main   # このブランチが触ったリンクだけ確認
npx drift verify --auto-approve       # staleをLLM判定し、整合なら自動再承認
npx drift coverage                    # どこまで仕様化できているかの数字
```

`coverage` はテストカバレッジのドキュメント版です。ただし100%を目指すべきではない(ツール自身も出力でそう言います)。すべてのヘルパー関数に仕様書は要らない。**公開APIと重要な振る舞いの一致が数字で追えること**に意味があります。

PR botはGitHub Actionsテンプレートをコピーするだけ。あえて**マージはブロックしません** — 導入初期に開発を止めるゲートは反感しか生まない。規律は強制ではなく、可視化から始まると考えています。

> **[画像10: drift coverage のスクリーンショット]**
> カラーのプログレスバー付き coverage 出力。kind別の内訳バーはビジュアルとして強い。

## 正直な現在地

強い言葉で書いてきたので、限界も同じ解像度で書きます。

- **実績はまだ自分のリポジトリだけです。** 11リンク・CI緑は「動作証明」であって「運用実績」ではありません。他人のチームの規模で回った数字は、これから作ります。
- **verifyの判定精度は未計測です。** 整合/乖離の正解データセットでの評価はこれから。数字が出るまで、`--auto-approve` は慎重に使ってください。
- **チーム運用にはまだ角があります。** リンクDBは現在SQLiteで、複数人が別ブランチで承認するとマージが痛い。テキストベースの追記型フォーマットへの移行を最優先の設計課題として認識しています。
- ソロ開発のv0.3です。思想に共感してくれた人の、最初の10人になってほしい — issueでもPRでも、「うちでは剥がれた」という報告でも、全部が今は貴重です。

それでも公開するのは、**この方向が正しいという確信は、自分のリポジトリが毎日証明してくれている**からです。

## 「動くはず」はテストになった。「合ってるはず」はまだ祈りのままだ

ソフトウェア開発の歴史は、祈りを仕組みに置き換えてきた歴史です。「動くはず」はテストになり、「壊してないはず」はCIになり、「デプロイできるはず」はCDになりました。

**「ドキュメントは合ってるはず」だけが、いまだに祈りのままです。**

私の答えは二段構えです。まず、**ドキュメントをコードと同じリポジトリに帰す**。歴史とレビューとコンテキストを一つにする。そしてその上に、**対応関係の機械的な証明**を張る。統合が土台で、証明がエンジン。どちらか片方では足りないことを、私は一度目の失敗(リポジトリ内で腐らせた)と二度目の失敗(Notionに出して遠ざけた)の両方で学びました。

仕様とコードの一致がテストカバレッジと同じ顔で数字になり、PRごとに機械が検証し、非エンジニアもエージェントも同じリポジトリを信頼して参照する。ドキュメントが「書いた瞬間から腐り始める負債」ではなく「コードと同じ規律で維持される資産」になる。

その世界は、運用ルールの徹底からは生まれません。**場所の統合と、仕組みから生まれます。**

```sh
npm i -D driftdocs && npx drift init
```
